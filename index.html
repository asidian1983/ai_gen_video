Problems Solved

단순 CRUD를 넘어선 문제 해결

Problem

AI 생성 작업이 수십 초~수 분 소요

Solution

BullMQ 비동기 잡 큐 + 지수 백오프 재시도 (3회)

Problem

동시 Worker의 Race Condition

Solution

DB 레벨 Optimistic Locking — UPDATE WHERE status=:expected

Problem

클라이언트 반복 폴링으로 인한 부하

Solution

Socket.IO WebSocket Room — EventEmitter2 도메인 이벤트 브리지

Problem

Solution

Dead Letter Queue — PostgreSQL 영구 저장 + 수동 재처리 API

Problem

API 남용 및 Brute-force 공격

Solution

3-tier Named Throttler — burst / standard / sustained 계층별 제한

Problem

AI 처리가 API 서버를 블로킹

Solution

API / Worker 컨테이너 분리 — 독립 스케일링 가능

One Image, Two Services

동일한 Docker 이미지를 API와 Worker 두 컨테이너로 실행. Worker는 독립 스케일링 가능.

# docker-compose.yml
api:
  image: ai-gen-video:latest
  command: node dist/main
  ports: ["3000:3000"]

worker:
  image: ai-gen-video:latest  # 동일 이미지
  command: node dist/main.worker
  # HTTP 포트 없음 — 독립 스케일링

Domain Event Fan-out

Processor, Gateway, DLQ가 직접 의존하지 않음. Kafka 마이그레이션 시 코드 변경 없음.

Deep Dive

핵심 설계 결정

Why를 설명할 수 있는 엔지니어링

Optimistic Locking

vs Pessimistic Lock (SELECT FOR UPDATE)

-- 단 한 줄로 Race Condition 차단
UPDATE videos
SET status = 'processing'
WHERE id = :id
  AND status = 'pending'

-- affected = 0 이면 다른 Worker가 선점
-- → 현재 Worker 조용히 종료

Pessimistic Lock은 트랜잭션 유지 중 네트워크 지연 시 전체 Worker 블로킹. Optimistic Lock은 시도 → 실패 감지 → 포기 패턴으로 불필요한 대기 없음.

EventEmitter2 도메인 이벤트

vs 직접 메서드 호출

// 직접 호출 (결합도 높음 ❌)
processor.videoGateway.emit()
processor.dlqService.capture()

// 도메인 이벤트 (결합도 낮음 ✅)
eventEmitter.emit('video.failed', event)
// Gateway, DLQ가 독립적으로 구독
// Kafka 교체 시 코드 변경 없음

새 소비자(Slack 알림 등) 추가 시 Processor 수정 불필요. in-process → Redis Pub/Sub → Kafka 마이그레이션이 이벤트 타입 계약만 유지하면 가능.
