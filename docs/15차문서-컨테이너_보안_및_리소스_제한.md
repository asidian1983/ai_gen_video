## 1. 개선 사항 요약

| 항목 | 이전 | 이후 |
|------|------|------|
| 권한 상승 방지 | 없음 | `security_opt: no-new-privileges:true` |
| 메모리 제한 | 없음 (컨테이너가 호스트 전체 소비 가능) | 서비스별 hard limit |
| CPU 제한 | 없음 | 서비스별 CPU 쿼터 |
| 리소스 예약 | 없음 | soft reservation으로 스케줄러 힌트 |
| OOM 보호 | 없음 | 메모리 limit → 초과 시 해당 컨테이너만 재시작 |

## 2. `no-new-privileges` — 왜 필요한가?

```yaml
security_opt:
  - no-new-privileges:true
```

Linux `PR_SET_NO_NEW_PRIVS` 플래그 설정. 컨테이너 내부 프로세스가 `execve()` 호출 시 **setuid/setgid 비트나 파일 capability로 권한을 상승시키지 못하게** 차단.

```
시나리오: 컨테이너 내부 취약점으로 공격자가 코드 실행 획득
→ no-new-privileges 없음: /usr/bin/sudo (setuid root) 실행 가능
→ no-new-privileges 있음: execve()에서 setuid 무시 → root 획득 불가
```

**Node.js API/Worker에 적합한 이유:**
- Node 프로세스는 권한 상승이 필요 없음 (PORT 3000은 비특권 포트)
- 이미 non-root 유저(`nestjs`, UID 1001)로 실행 중
- `no-new-privileges`는 이 상태를 execve 후에도 강제 유지

## 3. 리소스 제한 설계

### OCI Free Tier Ampere A1 스펙
- OCPU: 4 (ARM64 코어)
- RAM: 24 GB
- 총 4개 서비스 → 서비스당 균등 배분 + 여유분 확보

### 각 서비스별 배분

| 서비스 | memory limit | memory reservation | CPU limit | CPU reservation |
|--------|-------------|-------------------|-----------|----------------|
| api | 512m | 256m | 0.50 | 0.25 |
| worker | 512m | 256m | 0.50 | 0.25 |
| postgres | 512m | 256m | 0.50 | 0.25 |
| redis | 384m | 128m | 0.25 | 0.10 |
| **합계 limit** | **1.9 GB** | — | **1.75 CPU** | — |
| **여유** | **22 GB** | — | **2.25 CPU** | — |

**Redis를 더 작게 설정한 이유:**
- `--maxmemory 256mb`로 Redis 자체가 내부 메모리를 256mb로 제한
- 컨테이너 limit 384mb: Redis 프로세스 오버헤드 포함 여유분

### `limits` vs `reservations` 차이

```
limits (hard cap):
  컨테이너가 이 값을 초과하면 OOM killer 또는 CPU throttle
  → 다른 서비스에 영향을 주지 않도록 격리

reservations (soft hint):
  Docker 스케줄러에게 "이 정도는 항상 확보해줘"라는 힌트
  → 실제로 여유가 있으면 limit까지 사용 가능
```
