# 7차 문서 — 컨테이너 아키텍처 설계

> API / Worker / Redis 분리 컨테이너화 — 이미지 최적화 및 시작 시간 단축
> 작성일: 2026-03-18

---

## 1. 개요

### 변경 전 → 변경 후

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 서비스 분리 | API + Worker 동일 프로세스 | API 컨테이너 / Worker 컨테이너 분리 |
| Dockerfile 빌드 단계 | 2단계 (builder, production) | 3단계 (builder, production-deps, production) |
| 빌드 버그 | `npm ci --only=production` 후 `nest build` 실행 → devDeps 없어 빌드 불가 | builder에서 `npm ci` (전체) → production-deps에서 `--omit=dev` |
| Worker 진입점 | 없음 | `src/main.worker.ts` + `src/worker-app.module.ts` |
| `.dockerignore` | 없음 | 추가 (빌드 컨텍스트 최소화) |
| Redis 설정 | 기본값만 | `maxmemory 256mb`, LRU eviction, AOF 활성화 |

## 2. 컨테이너 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                    Docker Network (bridge)                   │
│                                                             │
│  ┌──────────────────┐      ┌──────────────────────────────┐ │
│  │   api:latest     │      │      worker:latest           │ │
│  │  node dist/main  │      │  node dist/main.worker       │ │
│  │  port 3000       │      │  no HTTP port                │ │
│  │  healthcheck:    │      │  healthcheck:                │ │
│  │  GET /health     │      │  kill -0 1                   │ │
│  └────────┬─────────┘      └───────────┬──────────────────┘ │
│           │                            │                     │
│           └──────────┬─────────────────┘                    │
│                      ▼                                       │
│         ┌────────────────────────┐                          │
│         │   Redis 7-alpine       │                          │
│         │   port 6379            │                          │
│         │   maxmemory 256mb LRU  │                          │
│         │   AOF + RDB snapshot   │                          │
│         └────────────────────────┘                          │
│                      │                                       │
│         ┌────────────────────────┐                          │
│         │   PostgreSQL 16-alpine │                          │
│         │   port 5432            │                          │
│         └────────────────────────┘                          │
└─────────────────────────────────────────────────────────────┘
```

**핵심 설계 원칙: One Image, Two Services**
- 동일한 Docker 이미지(`ai-gen-video:latest`)를 `api`와 `worker`가 공유
- `command`만 달라 코드/의존성 일관성 보장
- 이미지는 `api` 서비스에서 빌드 후 태깅 → `worker`는 재사용

## 3. Dockerfile — 3단계 멀티스테이지 빌드

```dockerfile
# Stage 1: builder — devDeps 포함 전체 설치 + TypeScript 컴파일
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                    # devDeps 포함 (nest build에 필요)
COPY . .
RUN npm run build             # → dist/ 생성

# Stage 2: production-deps — 프로덕션 의존성만 설치
FROM node:20-alpine AS production-deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force

# Stage 3: production — 최소 이미지
FROM node:20-alpine AS production
WORKDIR /app
RUN addgroup -g 1001 -S nodejs && adduser -S nestjs -u 1001
COPY --from=production-deps --chown=nestjs:nodejs /app/node_modules ./node_modules
COPY --from=builder         --chown=nestjs:nodejs /app/dist         ./dist
COPY --from=builder         --chown=nestjs:nodejs /app/package.json ./package.json
USER nestjs
ENV NODE_ENV=production
CMD ["node", "dist/main"]    # Worker는 docker-compose에서 override
```

### 이미지 최적화 포인트

| 최적화 | 방법 | 효과 |
|--------|------|------|
| devDeps 제거 | Stage 분리 (production-deps) | 이미지 크기 ~40% 감소 |
| 빌드 캐시 활용 | `package*.json` 먼저 COPY → `npm ci` → 소스 COPY | 소스 변경 시 재빌드 속도 향상 |
| npm 캐시 제거 | `npm cache clean --force` | 캐시 레이어 미포함 |
| 비루트 사용자 | `USER nestjs` | 보안 강화 |

### 기존 Dockerfile 버그 수정

```
# 기존 (버그): production deps만 설치 후 nest build 시도 → 실패
RUN npm ci --only=production   # @nestjs/cli 없음!
RUN npm run build              # Error: nest: command not found

# 수정: builder에서 전체 설치
RUN npm ci                     # devDeps 포함
RUN npm run build              # 성공
```

## 4. Worker 프로세스 분리

### `src/worker-app.module.ts`

API 전용 모듈(`AuthModule`, `ThrottlerModule`)을 제외한 헤드리스 NestJS 모듈:

```typescript
@Module({
  imports: [
    ConfigModule,   // 전역 설정
    WinstonModule,  // 로깅 (worker-combined.log, worker-error.log)
    TypeOrmModule,  // DB 쓰기 (Video 상태 업데이트)
    BullMQModule,   // 잡 큐 소비

    // Worker가 필요한 도메인 모듈만
    VideosModule,   // VideosService, VideoRepository
    AiModule,       // AiService (AI Provider 호출)
    QueueModule,    // VideoGenerationProcessor (BullMQ Worker)
    StorageModule,  // StorageService (S3 업로드)

    // 제외: AuthModule (JWT 불필요), ThrottlerModule (HTTP 없음)
  ],
})
export class WorkerAppModule {}
```

### `src/main.worker.ts`

```typescript
async function bootstrapWorker() {
  // createApplicationContext = HTTP 서버 없음
  const app = await NestFactory.createApplicationContext(WorkerAppModule);

  // Graceful shutdown (Docker stop → SIGTERM)
  process.on('SIGTERM', async () => {
    await app.close();  // BullMQ 연결 정리, 진행 중 잡 완료 후 종료
    process.exit(0);
  });
}
```

### API vs Worker 비교

| 항목 | API | Worker |
|------|-----|--------|
| 진입점 | `src/main.ts` | `src/main.worker.ts` |
| NestJS 부트스트랩 | `NestFactory.create()` | `NestFactory.createApplicationContext()` |
| HTTP 서버 | O (port 3000) | X |
| Swagger | O (dev only) | X |
| AuthModule | O | X |
| ThrottlerModule | O | X |
| QueueModule (processor) | O (동일 이미지) | O |
| 로그 파일 | combined.log | worker-combined.log |

## 5. docker-compose 서비스 구성

### 서비스 의존성 그래프

```
postgres (healthcheck: pg_isready)
  ↑
redis    (healthcheck: redis-cli ping)
  ↑
api      (healthcheck: GET /health)   ← 외부 포트 3000
  ↑
worker   (healthcheck: kill -0 1)     ← 포트 없음
```

### `api` 서비스

```yaml
api:
  build:
    context: .
    target: production
  image: ai-gen-video:latest   # 빌드 후 이 태그로 저장
  command: ["node", "dist/main"]
  healthcheck:
    test: ['CMD-SHELL', 'wget -qO- http://localhost:3000/health || exit 1']
    start_period: 15s          # DB 마이그레이션 시간 고려
```

### `worker` 서비스

```yaml
worker:
  image: ai-gen-video:latest   # api가 빌드한 이미지 재사용 (build: 없음)
  command: ["node", "dist/main.worker"]
  healthcheck:
    test: ['CMD-SHELL', 'kill -0 1 2>/dev/null || exit 1']
    # kill -0: 프로세스 존재 확인, 실제 kill 아님
```

### Redis 프로덕션 설정

```yaml
redis:
  command: >
    redis-server
    --requirepass ${REDIS_PASSWORD:-}
    --maxmemory 256mb           # OOM 방지
    --maxmemory-policy allkeys-lru  # 캐시 eviction
    --save 60 1                 # 60초마다 RDB 스냅샷
    --appendonly yes            # AOF 활성화 (내구성)
```

## 8. 운영 명령어

```bash
# 전체 스택 시작 (첫 실행 — 이미지 빌드 포함)
docker-compose up -d --build

# 로그 확인
docker-compose logs -f api
docker-compose logs -f worker

# Worker만 재시작 (코드 배포 시)
docker-compose up -d --build --no-deps worker

# 워커 수평 확장 (동일 큐에서 다수 소비)
docker-compose up -d --scale worker=3

# 전체 종료
docker-compose down

# 볼륨 포함 완전 초기화
docker-compose down -v
```

## 9. 수평 확장 전략

```
docker-compose up -d --scale worker=3

[worker_1] ─┐
[worker_2] ──┼─ BullMQ (Redis) ← VIDEO_GENERATION_QUEUE
[worker_3] ─┘
```

BullMQ는 Redis를 통해 잡을 분산 처리. 6차에서 구현한 Optimistic Locking으로 동일 잡을 여러 워커가 중복 처리하는 것을 방지.
